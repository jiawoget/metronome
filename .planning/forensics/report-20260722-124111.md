## Anomalies detected

### A1. Control-plane self-modification inside a product run — HIGH

**Evidence:** `gap-plan-repair-01` modifies both the product plan and the validator that decides whether the plan is live. Commits `86de4864` and `f2839e48` change the validator and its tests before R01 gap execution begins.

**Interpretation:** The run is testing and changing its own test harness. No checker result is comparable across rounds because the plan and/or checker protocol changed.

### A2. Native bounded revision loop bypassed — HIGH

**Evidence:** Native `plan-phase.md:1194-1267` owns a max-three revision loop and user escalation. The recorded gap run instead creates independent observability steps and agents. Its native `iteration_count` is never the authority for the sequence.

**Interpretation:** Splitting work into new step IDs resets the effective revision boundary. `retry_of` proves lineage but does not prove user approval or enforce a retry budget.

### A3. Numeric issue-count convergence misclassifies architectural churn — HIGH

**Evidence:** Native `plan-phase.md:1200-1219` detects a stall only when the number of issues does not decrease. The recorded substantive blocker count decreased `3 -> 2 -> 1`.

**Interpretation:** Different manifestations of one invalid transaction model appear as progress. The loop has no stable finding fingerprint, root-cause category, or “new architecture issue” detection.

### A4. Revision prompt structurally favors accretion — HIGH

**Evidence:** Native `plan-phase.md:1226-1247` instructs the planner to make targeted updates and not replan unless issues are explicitly fundamental. The checker emits local `suggested_fix` fields but no required root-cause class.

## Root-cause chain

The primary cause is not an individual missing plan field.

1. The architecture assigned CAP/reuse policy to planner/checker roles but did not place the project stop/retry policy in the controller that owns revision decisions. Repository policy now explicitly notes that skill injection does not reach every lifecycle actor.
2. The run then bypassed native plan-phase orchestration by manually creating independent checker/revision/repair steps.
3. Native issue-count convergence and the custom `retry_of` string check both treat locally decreasing or explicitly linked retries as valid progress; neither classifies architectural failure.
4. The planner is told to perform targeted edits, so a bad transaction model grows fields instead of being replaced.
5. The validator and its retry ledger are modified inside the same product run, invalidating comparability between checks.
6. Blocker identity is not durably recorded, preventing post-hoc detection of finding churn.

Compact statement:

> The product workflow became a self-modifying control system. The component deciding retries lacked project stop policy; the native bounded loop was bypassed; issue-count decline was mistaken for convergence; and observability enforced lineage without understanding blocker semantics. The resulting local-fix loop consumed the planning window while producing no product change.

## Repair criteria defined before hypothesis tests

> Historical design checkpoint: items 4-6 below were candidate controls written before independent review. Testing showed they would require unsupported extensions around the native checker loop. They were explicitly rejected and must not be implemented. The authoritative replacement is in **Implemented architecture repair**: native OpenGSD owns the loop, while only a control-plane self-modification finding terminates the product attempt.

The repair must be a replacement of ownership, not another allowlist entry:

1. Native OpenGSD must be the sole owner of plan revision, retry, promotion, and lifecycle state.
2. Product runs must not modify plan, policy, validator, or lifecycle tooling. A control-plane defect terminates the product run and becomes a separate workflow-debug incident.
3. Observability must return to measurement-only behavior; it must not decide plan liveness or retry legality.
4. A checker blocker must be classified as plan-local or process/architecture. Process/architecture blockers stop immediately and are recorded through native forensics; they are never fed into targeted plan revision.
5. At most one plan-local revision may occur before user escalation. A new blocker identity on recheck is not convergence.
6. Plan/check identity must bind the checked artifact bytes. Changing the plan starts a new identity; it cannot reuse the previous checker cache identity.
7. Project-folder-only execution must be stated at the controller and plan-checker seams; `use_worktrees=false` alone is insufficient.
8. Superseded custom artifacts and control code must be deleted or reduced after the replacement works; they must not coexist as a second active architecture.

Implementation began only after H1-H4 had reproducible tests or static proof and the removal boundary had been independently reviewed.

## Additional reliability failures exposed by tests

The first repair review found three measurement defects that would otherwise have hidden the same incident:

1. Sequential attempts with the same `(agent, step)` were folded into one long step. A RED test reproduced one 300-second step instead of 60-second and 180-second attempts; the summarizer now preserves both attempts.
2. An in-progress step used its start time as its end time and therefore reported zero wall time. A RED test reproduced the zero; summaries now measure the step and the whole run through generation time. The regression fixture holds one no-test step open for about 62 minutes and requires both `severe_over_budget` and `warning_no_test_change`, preventing the original stall window from being reported as `on_track`.
3. Dirty files created by another concurrent actor were automatically attributed as the current step's outputs. Output attribution is now declared-only.
4. Even after output attribution was fixed, the writer still ran `git status --untracked-files=all` plus `rev-parse HEAD` for every event and serialized the result. A RED test dirtied `.gitignore` and proved that hidden scan was still present. The Git snapshot and event field were deleted; the only remaining Git call resolves the repository root, so measurement no longer adds an undeclared full-worktree scan or captures another actor's state.

Missing token, tool, and external-wait data had also been encoded as zero. Token attribution is now `null` with `token_attribution_unavailable`; tool and external-wait values are `null` unless supplied. ## Implemented architecture repair

### Lifecycle ownership

- Native OpenGSD exclusively owns milestone, phase, plan checking, revision, retry, stall, execution, summary, verification, and escalation.
- Codex controllers bind native-resolved model and effort at typed subagent dispatch rather than relying on implicit inheritance; this is runtime binding, not a second lifecycle.
- A product attempt may not repair project policy, checking, observability, lifecycle tooling, or an unresolved environment prerequisite inside its own plan. Such a discovery terminates the product attempt and becomes separate forensics work.
- The repository no longer adds a stable-finding schema or a one-revision controller on top of OpenGSD; those unsupported extensions would have recreated the manual loop.
- `workflow.use_worktrees=false` is enforced as a project-folder-only contract at controller and policy seams. While historical linked worktrees remain, the controller also requires the current working directory, Git top-level, and primary `git worktree list --porcelain` entry to resolve to the same exact root before product-agent dispatch.

### Reuse and OSS governance

- Every production change still begins with local semantic duplicate search.
- The full installed-dependency, platform, and OSS waterfall now applies only when adding or replacing a generic capability, API, algorithm, shared surface, or dependency.
- Native `RESEARCH.md` holds reuse decisions and native `PLAN.md` holds integration design. The custom execution receipt, receipt SHA, four fingerprints, `pre_edit` lifecycle, and Approved Surface transaction were deleted.
- Semantic reindex is conditional on a proved stale/unhealthy index, not a per-agent default.
- Final PR review follows the same boundary; it no longer requires CAP chains, LOC, or Code Health for unrelated work, but a finding-free read-only `@codex` review of the exact final PR head remains mandatory and has no waiver.
- The one-off dormant-seed Git transaction moved to `skills/promoting-dormant-seeds/SKILL.md`; ordinary planner/executor/reviewer contexts no longer load it.

### Observability

- The writer is append-only measurement. It accepts event commands and declared step metadata, but no lifecycle command or retry/cache metadata.
- The writer does not inspect or serialize worktree state; inputs and outputs are controller-declared only.
- The summarizer reports sequential attempts, in-progress step/run wall time, declared I/O, soft wall budgets, and explicit measurement limits.
- When the controller declares them, compact summaries also report agent type, model, and reasoning effort so future latency comparisons are attributable.
- `fingerprint`, `validate-plan`, and rollout-token attribution are rejected rather than silently ignored.
- Repository npm scripts provide the documented local runtime path, avoiding the previously repeated system-Node environment failure.
- The repository-local PowerShell npm wrapper now preserves JSON stdin, so declared I/O and the required local Node runtime no longer conflict.
